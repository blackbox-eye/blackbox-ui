# AGENTS.md — blackbox-ui

## Purpose

This repository is used for scoped frontend, governance, and documentation maintenance tasks on the Blackbox EYE public website and its canonical operating docs.

This file defines the mandatory operating contract for AI coding agents working in this repository.

Agents must keep tasks small, isolated, testable, reversible, and reviewable.

The primary goal is to preserve the live public design and user-facing experience while reducing technical debt safely.

---

## Workflow governance lock

This repo uses [docs/governance/BLACKBOX_UI_WORKFLOW_LOCK.md](docs/governance/BLACKBOX_UI_WORKFLOW_LOCK.md) as canonical ChatGPT, Copilot, and user workflow governance.

[docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md](docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md) defines the standard PR package, QA, and merge-readiness templates.

Copilot must stop if a task prompt lacks:

- allowed files
- forbidden files
- output required
- stop conditions
- branch or PR intent
- owner approval boundary

Copilot must not compensate for missing governance by guessing.

If the prompt is under-scoped, Copilot must report `handoff incomplete` instead of editing files.

---

## Golden rules

- One task = one branch = one pull request.
- Do not change `main` directly.
- Do not reuse an old branch or old PR for a new task unless explicitly instructed.
- Do not expand scope.
- Do not perform unrelated cleanup.
- Do not perform broad refactors.
- Do not make visual/design changes unless the task explicitly requests them.
- Do not add helper scripts unless explicitly requested.
- Do not commit temporary diagnostic scripts.
- Do not touch secrets, environment files, deployment files, runtime behavior, or hosting configuration unless explicitly approved.
- Do not change dependencies, `package.json`, or lockfiles unless the task explicitly requires it.
- Do not touch generated or minified files directly unless the source/build contract is already known and the task explicitly allows it.

If the root cause is unclear, stop and report.

---

## Task category model

Every task must declare one primary category before work starts:

1. docs governance
2. read-only audit
3. UI/bug
4. security
5. deploy/governance
6. legal/claims governance
7. build/source contract

If the task does not fit one primary category, or if it mixes categories without explicit owner approval, stop and ask for split scope.

---

## Required pre-code plan

Before editing files, produce a short plan containing:

1. task category
2. root-cause hypothesis
3. exact files to inspect
4. exact files expected to change
5. exact files forbidden for this task
6. source/generated asset impact
7. acceptance criteria
8. tests/checks to run
9. rollback risk
10. stop conditions

Do not start coding until the plan is approved, unless the task explicitly says execution is already approved.

For complex UI, navigation, scroll, build, or generated-asset tasks, use Interactive Plan or Review mode. Do not use raw Start.

---

## Scope discipline

Only edit files directly required by the task.

If the requested fix appears to require touching more than five files, stop and report:

- why more files are needed
- which files
- whether the task should be split into smaller PRs

No “while here” changes.

No silent refactors.

No cleanup outside the approved task.

## Source/generated asset contract

Generated and minified files must not be patched manually unless the task explicitly allows it and the source/build path is known.

Known build commands from `package.json`:

- `npm run build:tailwind`
- `npm run build:css`
- `npm run build`
- `npm test`
- `npm run test:raw`
- `npm run test:ci`

Known source/generated relationships:

- `assets/css/tailwind-src.css` is the Tailwind source input.
- `assets/css/tailwind.full.css` is generated by `npm run build:tailwind`.
- `assets/css/marketing.css` is source.
- `assets/css/marketing.min.css` is generated by `npm run build:css`.
- `assets/css/admin.css` is source.
- `assets/css/admin.min.css` is generated by `npm run build:css`.

Unclear until explicitly audited:

- `assets/js/site.js`
- `assets/js/site.min.js`
- the exact JavaScript minification/build command

Until the JavaScript build contract is documented:

- Do not edit `assets/js/site.min.js` directly.
- Do not claim JavaScript-generated artifacts are correct unless the source/build path is verified.
- If a JS fix requires minified output, stop and report the missing build contract.

## CSS rules

CSS changes must be minimal and local to the issue.

Prefer:

- one state contract
- one source of truth
- smallest conflicting rule fix
- scoped selectors
- predictable cascade
- existing tokens/classes when available

Avoid:

- global `!important` dumps
- duplicate state systems
- inline style enforcement from JavaScript
- broad visual rewrites
- unrelated design cleanup
- manual edits to generated CSS
- patching multiple component layers for the same state

If generated CSS must change, state:

1. source file edited
2. build command run
3. generated file produced
4. why generated output is included

---

## JavaScript rules

JavaScript may toggle classes and ARIA state.

JavaScript must not force layout with inline styles unless explicitly approved.

Preserve accessibility state where relevant:

- `aria-expanded`
- `aria-hidden`
- `aria-controls`
- focus behavior
- keyboard access
- escape key behavior

Avoid:

- duplicate event handlers
- competing state logic
- mixed open-state systems
- layout fixes implemented through inline styles
- manual patches to minified JavaScript

If JS ownership is unclear, stop and report before editing.

---

## UI state contract rules

Every interactive component must have one clear state contract.

A component contract must define:

1. state class
2. ARIA state
3. JavaScript owner
4. CSS owner
5. markup owner
6. test file
7. rollback risk

Do not mix multiple state systems unless compatibility is explicitly required.

Avoid mixing:

- `.active`
- `.is-open`
- `.is-visible`
- `[aria-hidden="false"]`
- `[data-open="true"]`
- inline `display`
- inline `transform`
- Tailwind transform utility overrides

---

## Mobile navigation / burger menu contract

For mobile drawer or burger menu fixes, prefer this single open-state contract:

- `#mobile-menu.active`
- `#mobile-menu-overlay.active`
- `body.mobile-menu-open` only if already required by existing code

ARIA should be accessibility state, not the primary CSS-open contract:

- burger button: `aria-expanded="true|false"`
- drawer: `aria-hidden="true|false"`
- overlay: `aria-hidden="true|false"`

Do not use `[aria-hidden="false"]` as the main CSS open selector unless the approved task explicitly requires legacy compatibility.

For burger menu work, verify:

- menu is hidden on initial load
- burger opens menu
- overlay becomes visible and clickable
- close button closes menu
- overlay click closes menu
- menu links are clickable
- desktop nav still works
- no header clipping
- no scroll-lock regression

---

## Header / navigation rules

When touching header or navigation, preserve:

- desktop navigation
- mobile burger visibility
- language switch behavior
- theme toggle behavior
- console/login dropdown behavior unless explicitly out of scope
- header height stability
- breadcrumbs visibility
- scroll behavior
- z-index/overlay hierarchy

Do not redesign navigation unless explicitly requested.

---

## Scroll and overlay rules

Scroll-lock changes are high risk.

Do not touch scroll-lock logic unless the task explicitly names scroll behavior.

When touching drawer, overlay, modal, assistant, or sticky elements, verify:

- body remains scrollable when closed
- body locks only when intended
- no stale lock classes remain
- overlay does not intercept clicks when hidden
- z-index does not block header/nav/assistant unexpectedly

Do not introduce global scroll fixes as part of a component task.

---

## UI/runtime test requirements

Before marking a UI, runtime, or behavior-affecting PR ready, run the smallest relevant test set.

Docs governance, legal/claims governance, deploy/governance docs, and read-only audit tasks should follow the docs-only validation rules in [docs/governance/BLACKBOX_UI_WORKFLOW_LOCK.md](docs/governance/BLACKBOX_UI_WORKFLOW_LOCK.md) and the matching template in [docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md](docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md) instead of these UI checks.

For UI/header/mobile navigation work, validate at minimum:

- 320px mobile viewport
- 390px mobile viewport
- 768px tablet viewport
- 1024px desktop threshold
- desktop width above 1280px when desktop navigation is affected

For small UI fixes, do not run broad unrelated suites unless required by CI.

If a narrow test does not exist, either:

- add one narrow test if approved, or
- report that no narrow test exists and stop if the task requires verification

If tests fail twice, stop and report:

1. exact command
2. exact failing assertion
3. suspected owner file
4. whether failure is related or unrelated
5. recommended next isolated task

## PR package requirements

Every PR must follow the canonical package in [docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md](docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md). Use the template that matches the task category.

For UI/bug PRs, include target pages and viewports validated, source/generated asset impact when relevant, and what was intentionally not changed.

Docs governance, security, deploy/governance, legal/claims governance, and read-only audit work must not be forced into the UI-specific PR package.

Do not mark a PR ready if the acceptance criteria, validation boundary, scope guard, or owner approval boundary are unresolved.

Do not claim the issue is fixed unless the validated evidence supports that claim.

## Required final response format

Follow the required output in the active task prompt when one exists.

Otherwise, use the completion report and PR package structure defined in [docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md](docs/governance/BLACKBOX_UI_PR_PACKAGE_AND_QA_SOP.md).

Do not default to the UI-specific PR/output format for docs governance, security, deploy/governance, legal/claims governance, or read-only audit work unless the task prompt explicitly requires it.

## Read-only audit task rules

For read-only audit tasks:

- Do not edit files.
- Do not create a PR.
- Do not change docs.
- Do not run refactors.
- Do not fix bugs.
- Return findings only.

Required audit output:

1. executive summary
2. scope and mode
3. confirmed findings
4. open questions and caveats
5. risks or blockers
6. recommended next tasks
7. stop conditions

Specialized audit schemas belong in canonical repo docs or the active task prompt. Use them only when they are explicitly required.

---

## Stop rules

Stop and return for review if:

- root cause is unclear
- source/generated ownership is unclear
- JS ownership is unclear
- the fix needs broad refactor
- more than five files are required
- tests fail twice
- the task conflicts with this AGENTS.md
- unrelated files are being pulled into the diff
- generated artifacts are unclear
- minified files need manual edits
- package files need changes not approved in scope
- the branch becomes polluted with temporary scripts
- the task requires repo restructuring
- the task risks changing user-facing design unexpectedly

## Branch and PR naming

Use clear names.

Branch examples:

- `fix/mobile-burger-open-state`
- `fix/header-breadcrumb-spacing`
- `fix/desktop-more-dropdown`
- `test/mobile-nav-smoke`
- `docs/frontend-build-contract`
- `docs/ui-state-contracts`
- `audit/frontend-build-flow`

PR title examples:

- `fix(ui): mobile burger menu open state`
- `fix(ui): header spacing prevents breadcrumb clipping`
- `test(ui): add mobile navigation smoke coverage`
- `docs(ui): define frontend build contract`
- `docs(ui): define component state contracts`

---

## Human review rule

A passing CI run is required but not sufficient.

Do not merge or recommend merge unless:

- the target behavior is validated
- the changed files are listed
- generated asset impact is explained
- the diff is scoped
- the acceptance criteria are satisfied
- no unresolved review comments remain
- human review is complete
